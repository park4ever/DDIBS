# DDIBS V1 Final Review

## 1. 문서 목적

이 문서는 DDIBS V1의 구현 범위, 핵심 플로우, 설계 판단, 테스트 결과, 한계와 다음 단계(V2 후보)를 정리한 마감 문서다.

`DDIBS_MASTER_BLUEPRINT_v1.1.md`가 프로젝트의 기준 문서라면, 이 문서는 **V1에서 실제로 어디까지 구현했고 무엇을 검증했는지**를 정리하는 결과 문서다.

---

## 2. 프로젝트 한 줄 요약

DDIBS는 한정 수량 상품 발매 상황에서 **주문 생성 시 재고 홀드**, **홀드 만료 시 자동 해제 및 재고 복구**, **결제 성공/실패에 따른 상태 전이**, **확정 주문 기준 정산 생성**까지를 안정적으로 처리하는 Java/Spring 백엔드 드롭 런치 플랫폼이다.

---

## 3. V1 목표

V1의 핵심 목표는 기능 수를 늘리는 것이 아니라, 아래 문제를 백엔드 관점에서 설명 가능하게 해결하는 것이었다.

* 발매 가능 여부 검증
* 주문 생성 시 재고 홀드
* 결제 전 임시 선점 유지
* 결제 실패 또는 홀드 만료 시 자동 해제 및 재고 복구
* 동시 주문 상황에서 과판매 방지
* 확정 주문 기준 정산 생성

즉 DDIBS V1은 **정합성, 동시성 제어, 상태 전이 보호**를 실제 코드와 테스트로 증명하는 데 집중했다.

## 5. 핵심 플로우

### 5.1 플로우 1: 주문 생성 → 재고 홀드 → 결제 → 주문 확정 → 정산 생성

```text
발매 조회
→ 발매 가능 여부 검증
→ 주문 생성 (Order.CREATED)
→ 홀드 생성 (HoldReservation.ACTIVE)
→ LaunchVariant.availableStock 감소
→ 결제 요청 (Payment.PENDING)
→ 결제 성공
→ Order.CONFIRMED / HoldReservation.CONSUMED / Payment.SUCCESS
→ 정산 생성 배치
→ Settlement.CREATED
```

### 5.2 플로우 2: 홀드 만료 배치 → 주문 만료 → 재고 복구

```text
만료 대상 HoldReservation 조회
→ HoldReservation.EXPIRED
→ Order.HOLD_EXPIRED
→ LaunchVariant.availableStock 복구
```

## 7. V1에서 중요했던 설계 판단

### 7.1 실제 재고 차감 단위는 LaunchVariant

DDIBS는 상시 판매 재고가 아니라 **이번 발매에서 실제로 열리는 판매 단위**를 중심으로 재고를 관리한다.

* `ProductVariant`: 판매 단위의 원형
* `LaunchVariant`: 이번 발매의 실제 판매/재고 단위

따라서 주문 가능 여부, 재고 차감, 재고 복구는 `LaunchVariant`가 책임진다.

### 7.2 주문 생성 시 홀드를 함께 만든다

주문과 홀드를 분리된 후처리로 두지 않고, 주문 생성 시점에 바로 홀드를 생성했다.

* `Order.CREATED`
* `HoldReservation.ACTIVE`
* `availableStock` 감소
* `expiresAt` 설정

이를 통해 결제 전 임시 선점 상태를 명확히 표현했다.

### 7.3 결제와 정산 생성을 분리했다

결제 성공 시점에는

* Payment 성공 처리
* Order 확정
* Hold 소비 처리

까지만 수행하고, 정산은 **별도 배치**가 `CONFIRMED` 주문을 기준으로 생성하도록 설계했다.

이렇게 분리해서 결제 플로우의 책임을 줄이고, 정산 생성의 운영 성격을 더 분명히 했다.

### 7.4 홀드 만료도 배치로 분리했다

홀드 TTL이 지난 대상을 주기적으로 찾고,

* Hold 만료
* Order 만료
* 재고 복구

를 자동화하는 별도 배치로 구성했다.

## 8. 동시성 처리 방식

### 8.1 주문 생성 경합

주문 생성 시 `LaunchVariant`를 **비관적 락(PESSIMISTIC_WRITE)** 으로 조회해, 재고 1건 상황에서 동시에 여러 요청이 들어와도 1건만 성공하도록 처리했다.

핵심 목표는 아래 비즈니스 결과를 보장하는 것이었다.

* 과판매가 발생하지 않는다
* 재고 1건 상황에서 오직 1건만 홀드 생성에 성공한다
* 실패한 주문은 재고를 점유하지 못한다

### 8.2 결제 vs 홀드 만료 배치 경합

V1 후반 점검에서, **결제 성공 요청과 홀드 만료 배치가 거의 동시에 실행될 때** 상태와 재고가 섞일 수 있는 문제가 발견되었다.

실제 테스트에서 아래와 같은 모순 상태가 확인되었다.

* Payment `SUCCESS`
* Order `CONFIRMED`
* Hold `CONSUMED`
* 그런데 재고는 복구됨

이 문제를 해결하기 위해 다음과 같이 리팩토링했다.

* `Order` 락 조회 메서드 추가
* `HoldReservation` 락 조회 메서드 추가
* `PaymentService`는 결제 시작 시 `Order`와 `HoldReservation`을 **락 + 최신 상태 기준**으로 재검증
* `HoldExpirationBatchService`는 만료 후보를 바로 처리하지 않고, **후보 ID 조회 → 락 재조회 → 최신 상태 검증** 구조로 변경
* 두 서비스 모두 같은 락 순서를 따르도록 정리

이후 경합 테스트를 다시 실행해, 최종 상태가 **정책적으로 일관된 두 종결 상태 중 하나로만 수렴함**을 확인했다.

## 9. 테스트 전략

V1에서는 단순 happy path보다, **실패 / 복구 / 경합** 시나리오를 함께 검증하는 테스트 전략을 선택했다.

### 9.1 주요 테스트 대상

* `OrderServiceIntegrationTest`

    * 주문 생성 성공
    * 주문 불가 상태 실패
    * 재고 부족 실패

* `PaymentServiceIntegrationTest`

    * 결제 성공
    * 결제 실패
    * 이미 결제가 완료된 주문 재결제 차단

* `SettlementBatchServiceIntegrationTest`

    * 확정 주문 정산 생성
    * 중복 정산 생성 방지
    * 미확정 주문 제외

* `HoldExpirationBatchServiceIntegrationTest`

    * 만료된 ACTIVE 홀드 처리
    * ACTIVE가 아닌 홀드 제외
    * 아직 만료되지 않은 홀드 제외

* `OrderConcurrencyIntegrationTest`

    * 재고 1개 상황 동시 주문 시 1건만 성공

* `PaymentHoldExpirationRaceIntegrationTest`

    * 결제 성공 요청 vs 홀드 만료 배치 경합 시 일관된 최종 상태로 수렴하는지 검증

### 9.2 테스트 관점 요약

V1 테스트는 아래를 증명하는 데 초점을 맞췄다.

* 상태 전이가 허용된 흐름으로만 진행되는가
* 실패 시 재고가 정확히 복구되는가
* 배치가 중복 생성 / 중복 복구를 일으키지 않는가
* 경합 상황에서 최종 상태가 섞이지 않고 일관되게 수렴하는가

## 10. V1에서 해결한 문제

### 10.1 주문 생성 시 재고 홀드

주문 생성과 동시에 홀드를 만들고, 결제 전 일정 시간 동안 재고를 임시 선점하도록 구현했다.

### 10.2 결제 실패 / 홀드 만료 시 재고 복구

재고 복구가 단순 보조 로직이 아니라, **주문 실패 / 홀드 만료와 함께 묶여야 하는 핵심 복구 시나리오**임을 코드로 반영했다.

### 10.3 정산 생성 배치

정산을 결제 성공 시점에 직접 생성하지 않고, 확정 주문 기준으로 별도 배치가 생성하도록 분리했다.

### 10.4 주문 생성 동시성 제어

재고 1개 상황 동시 주문에서 1건만 성공하고 과판매가 발생하지 않음을 테스트로 검증했다.

### 10.5 결제와 만료 배치 경합 정리

V1 최종 점검 과정에서 실제 경합 버그를 발견했고, 이를 서비스/리포지토리 리팩토링과 경합 테스트로 해결했다.

## 11. V1의 한계

V1은 핵심 플로우 완성도에 집중한 버전이므로, 아래는 의도적으로 단순화했거나 추후 확장 대상으로 남겨두었다.

* 실제 PG 연동 없음 (모킹 결제)
* 정산 계산 로직 단순화
* 관리자 조회 조건/검색 기능 제한적
* 배치 실행 보장(예: 분산 환경 scheduler lock) 고도화 미적용
* Testcontainers 기반 테스트 환경 미적용
* 운영 로그/모니터링/알림 고도화 미적용

즉 V1은 운영 복잡도를 과도하게 늘리기보다, **핵심 비즈니스 결과를 안정적으로 증명하는 것**을 우선했다.

## 13. 회고

DDIBS V1에서 가장 큰 수확은 단순 CRUD 기능을 많이 넣는 것이 아니라,

* 주문 생성 시 재고를 어떻게 임시 선점할 것인가
* 결제 실패와 홀드 만료를 어떻게 복구 시나리오로 다룰 것인가
* 정산 생성을 왜 결제 흐름과 분리해야 하는가
* 동시성/경합 상황에서 상태와 재고 정합성을 어떻게 지킬 것인가

를 코드와 테스트로 설명 가능하게 만든 점이다.

특히 V1 마감 과정에서 발견한 **결제 vs 홀드 만료 배치 경합 버그**는 DDIBS의 정체성을 더 분명히 해준 이슈였다. 이 문제를 테스트로 재현하고, 락/재검증 구조로 해결하면서, DDIBS는 단순한 쇼핑몰 백엔드가 아니라 **고수요 발매 처리 시스템**으로서의 성격을 더 명확히 갖추게 되었다.

## 14. 한 줄 결론

DDIBS V1은 한정 수량 발매 상황에서 **주문, 재고 홀드, 결제, 홀드 만료 복구, 정산 생성**까지의 핵심 플로우를 완성하고, **정합성 · 동시성 · 상태 전이**를 테스트로 증명한 첫 번째 완성 버전이다.
