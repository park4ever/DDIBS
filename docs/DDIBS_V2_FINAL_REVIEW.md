# DDIBS V2 Final Review

## 1. 문서 목적

이 문서는 DDIBS V2에서 수행한 개선 사항과 그 의미를 정리한 마감 문서다.  
V1이 **핵심 플로우 완성**에 집중한 단계였다면, V2는 그 위에 **테스트 재현성**, **배치 운영성**, **관리자 조회**, **테스트 구조 정리**를 더해 프로젝트를 한 단계 더 실무적으로 다듬는 데 집중했다.

즉 이 문서는 “V2에서 무엇을 왜 바꿨는가”를 설명하는 문서다.

---

## 2. V2 목표

V2의 목표는 새로운 기능을 무작정 추가하는 것이 아니었다.  
DDIBS가 이미 가진 핵심 흐름을 더 믿을 수 있고, 더 설명 가능하며, 더 운영 가능한 구조로 만드는 것이 V2의 핵심 목표였다.

정리하면 V2 목표는 아래와 같았다.

- 테스트 환경을 로컬 의존에서 분리하고 재현성을 높인다.
- 배치 로직의 결과를 더 구조적으로 표현하고 운영 로그를 개선한다.
- 관리자 관점의 주문 조회 기능을 추가해 운영 추적성을 높인다.
- 테스트 코드를 정리해 프로젝트 전체 품질을 높인다.

---

## 3. V2 완료 항목

### 3.1 Testcontainers + MySQL 테스트 환경 정리

V1에서는 통합 테스트가 로컬 MySQL 환경에 일부 의존하는 형태였다.  
V2에서는 이를 **Testcontainers + MySQL + Flyway** 기반으로 전환했다.

주요 개선 내용:
- 테스트 실행 시 컨테이너 기반 MySQL 자동 기동
- Spring Boot `@ServiceConnection` 기반 연결 구성
- 테스트에서 `Flyway + ddl-auto=validate` 조합 사용
- 로컬 테스트 DB 준비 부담 제거
- 테스트 환경 재현성 강화

이 변경으로 DDIBS 테스트는 “내 컴퓨터에서만 되는 테스트”에서 벗어나,  
보다 **독립적이고 신뢰 가능한 통합 테스트 환경**에 가까워졌다.

---

### 3.2 정산 배치 멱등성 및 운영 로그 개선

정산 배치는 기존에도 동작했지만, V2에서는 **왜 안전한지 설명 가능한 구조**로 다듬는 데 집중했다.

주요 개선 내용:
- 확정 주문 전체 조회 후 exists 체크하는 방식에서,
  **미정산 확정 주문 후보 조회** 중심 구조로 개선
- `SettlementBatchResult` 추가
- 후보 수 / 생성 수 / 경쟁 스킵 수 구조화
- 스케줄러 로그 개선
- 관련 통합 테스트도 결과 구조 기준으로 보강

핵심 의미:
- 멱등성 설명력이 좋아짐
- 중복 생성 방지 방식이 더 명확해짐
- 배치 로그만 봐도 무엇이 일어났는지 해석하기 쉬워짐

---

### 3.3 홀드 만료 배치 결과 구조화 및 운영 로그 개선

홀드 만료 배치도 정산 배치와 같은 철학으로 정리했다.  
핵심 정합성 로직은 유지하면서, **운영 관점 결과를 더 명확하게 드러내는 방향**으로 리팩토링했다.

주요 개선 내용:
- `HoldExpirationBatchResult` 추가
- 후보 수 / 실제 만료 수 / 주문 상태 스킵 수 / 홀드 상태 스킵 수 구조화
- 스케줄러 로그 보강
- 관련 통합 테스트를 결과 구조 기준으로 정리
- 스케줄러 이름도 `HoldExpirationBatchScheduler`로 통일

핵심 의미:
- 배치가 단순히 “몇 건 처리했다”를 넘어서,
  **왜 스킵됐는지까지 설명 가능한 구조**가 됨
- 운영자가 배치 동작을 이해하고 추적하기 쉬워짐

---

### 3.4 관리자 주문 조회 API 추가

V2에서는 운영자 관점의 조회 기능도 보강했다.  
특히 주문은 DDIBS의 핵심 도메인이므로, **관리자 주문 조회 API**를 우선적으로 구현했다.

주요 개선 내용:
- QueryDSL 도입
- `AdminOrderSearchRequest`, `AdminOrderSummaryResponse` 추가
- `OrderRepositoryCustom`, `OrderRepositoryImpl` 구현
- `AdminOrderQueryService`, `AdminOrderController` 추가
- 검색 조건 + 페이징 + 정렬 지원
- `AdminOrderQueryServiceIntegrationTest` 작성

지원 조건:
- 주문 코드
- 주문 상태
- 판매자 ID
- 회원 ID
- 회원 이메일 키워드
- 상품명 키워드
- 기간(from / to)
- 페이징 / 정렬

핵심 의미:
- DDIBS가 단순히 핵심 흐름만 있는 시스템이 아니라,
  **운영자가 실제로 주문을 탐색하고 추적할 수 있는 시스템**으로 확장됨

---

### 3.5 테스트 구조 정리 및 핵심 시나리오 검증 보강

V2 후반에는 테스트 코드를 한 번 더 정리했다.  
이 단계는 새 기능 추가보다, 이미 있는 테스트가 **더 설명 가능하고, 더 일관성 있게 읽히도록** 만드는 데 집중했다.

주요 개선 내용:
- 고정 식별자 사용을 `uniqueSuffix()` 기반으로 정리
- fixture 생성 방식 일관성 보강
- 주문/결제/경합 테스트의 가독성 개선
- 동시성 테스트에서 기대 실패와 예상 밖 예외를 더 명확히 구분
- 경합 테스트에서 허용 가능한 종결 상태를 더 분명하게 표현
- 배치 테스트의 일관성 보정

핵심 의미:
- 테스트 자체가 프로젝트 설명 자료로서 더 좋아짐
- 정합성/동시성/경합 정책이 테스트 코드에 더 잘 드러남

## 4. V1 대비 달라진 점

### V1
- 핵심 플로우 완성
- 주문 생성 / 홀드 / 결제 / 정산 / 홀드 만료
- 동시 주문 방지
- 결제 vs 홀드 만료 배치 경합 처리
- 핵심 테스트와 README 정리

### V2
- 테스트 환경 재현성 강화
- 배치 결과 구조화 및 운영 로그 개선
- 관리자 주문 조회 API 추가
- 테스트 구조 정리 및 설명 가능성 강화

즉 V1이 “핵심 흐름을 만들고 증명한 단계”였다면,  
V2는 “그 흐름을 더 실무적으로 다듬고 운영 가능한 형태로 정리한 단계”라고 볼 수 있다.

---

## 5. 이번 단계에서 중요했던 설계 판단

### 5.1 테스트 환경은 실제 MySQL에 가깝게
DDIBS는 락, 경합, 배치, 상태 전이 같은 요소가 중요하므로,  
테스트도 실제 MySQL에 가까운 환경에서 검증하는 것이 중요했다.

그래서 V2에서 Testcontainers + MySQL 조합으로 전환한 것은,  
단순 편의 개선이 아니라 **프로젝트 정체성과 맞는 테스트 전략 강화**였다.

### 5.2 배치는 처리 건수만이 아니라 “결과 구조”가 중요하다
정산 생성 배치와 홀드 만료 배치는 모두 운영 관점이 강하다.  
따라서 단순히 `int count`만 반환하는 구조보다,  
**후보 / 성공 / 스킵**을 구분하는 구조가 더 설명 가능하고 실무적이라고 판단했다.

### 5.3 관리자 조회는 QueryDSL이 더 적합하다
관리자 주문 조회는 동적 조건이 많고, 페이징/정렬/검색 조합이 중요하다.  
따라서 파생 메서드나 단순 `@Query`보다, **QueryDSL 기반 커스텀 조회**가 더 자연스럽고 확장 가능하다고 판단했다.

### 5.4 테스트 정리는 “보여주기”가 아니라 “설명력 강화”다
V2에서 테스트 구조를 다듬은 이유는 단순히 예쁘게 만들기 위해서가 아니라,  
DDIBS의 핵심 정책이 테스트에 더 잘 드러나도록 하기 위해서였다.

---

## 6. V2 기준 현재 강점

V2를 마친 현재 DDIBS의 강점은 아래와 같다.

- 핵심 플로우 2개가 완성되어 있다.
- 상태 전이가 도메인 정책으로 보호된다.
- 동시 주문 과판매 방지를 테스트로 검증했다.
- 결제 vs 홀드 만료 배치 경합을 통합 테스트로 검증했다.
- 배치 결과가 구조화되어 운영 관점 설명력이 좋아졌다.
- 관리자 주문 조회 API로 운영 추적성이 보강되었다.
- 테스트가 Testcontainers 기반으로 재현 가능하게 정리되었다.

즉 DDIBS는 단순 CRUD 커머스가 아니라,  
**고수요 발매 상황을 정합성/동시성/운영성 중심으로 다룬 백엔드 시스템**이라는 메시지가 더 분명해졌다.

---

## 7. 아직 남아 있는 한계

V2까지 진행했어도, 아직 의도적으로 남겨둔 범위는 있다.

- 관리자 정산 조회/API는 아직 최소 수준
- 발매/재고 운영 조회는 추가 보강 여지 있음
- 실제 PG 연동 없음
- 프론트엔드 데모 레이어 없음
- 시퀀스 다이어그램 / 운영 문서 보강 여지 있음
- 운영 알림/관측 포인트는 아직 최소 수준

즉 V2는 완결이라기보다,  
**V3로 넘어가기 전에 DDIBS를 실무적인 형태로 한 번 다듬은 단계**라고 보는 것이 맞다.

---

## 8. V3 후보

다음 단계에서 우선적으로 고려할 수 있는 후보는 아래와 같다.

- 관리자 정산 조회/API 고도화
- 발매/재고 운영 조회 고도화
- 문서 및 시퀀스 다이어그램 보강
- 프론트엔드 데모 레이어 추가
- 운영 로그/관측 포인트 보강

이 역시 방향은 명확하다.  
일반 커머스 기능을 계속 늘리기보다, **DDIBS의 정체성을 더 강하게 보여주는 방향**으로 가는 것이 맞다.

---

## 9. 한 줄 결론

DDIBS V2는 핵심 플로우를 새로 만드는 단계가 아니라,  
**테스트 재현성, 배치 운영성, 관리자 조회, 테스트 설명 가능성**을 보강해  
프로젝트를 더 실무적으로 다듬은 단계였다.
